# Output Format Reference

Two distinct formats depending on modality. Read the relevant section before generating.

---

## FORMAT A — ON-SITE (Printable / Annotatable)

### Design Principles
- **Purpose:** Printed and used during live sessions. The facilitator writes by hand during the interview.
- **Whitespace:** Generous. Every answer zone needs room to write. Use `_______________` underlines or blank table cells.
- **Script:** Facilitator instructions in *italics*. What to say out loud in "quotes".
- **Tasks:** Use a structured table with columns for observation and notes.
- **Checkboxes:** Use ☐ for binary items the facilitator ticks during the session.
- **Tone:** Neutral and functional. No excessive emojis in the printable sections.
- **Font guidance:** Note "Print in A4, font size minimum 11pt" at the top.

### On-site Writing Rules
- Every question needs a blank notes area of at least 3 lines
- The scenario is written in a box / bordered section to make it visually distinct from tasks
- Task table must always have 3 columns: Échec / Hésitant / Succès (checkboxes)
- Pre-test questions should be warm, open, conversational (not Yes/No)
- The facilitator script (what to say) is always in "quotes" to distinguish from instructions

## FORMAT B — REMOTE (Digital / UserTesting)

### Design Principles
- **Purpose:** Read on screen by a remote participant or used by a facilitator on video call.
- **Tone:** Warm, welcoming, slightly playful. Emojis welcome.
- **Structure:** Clear phases with estimated timing. Each phase has a goal.
- **Script:** Conversational. Questions are written as the facilitator would actually say them.
- **Length:** Aim for 45–60 min total. Indicate time per phase.
- **Screener:** Full UserTesting questions with Accept/Reject labels (see screener-templates.md).

### Remote Writing Rules
- Phase timing should add up to 45–65 minutes total
- Every phase must have an explicit "Objective" line
- Scenario text (before tasks) must be in quotes and read aloud
- Post-task questions should probe: What did you expect? What was confusing? What would help?
- The participant welcome script ("Hello! 👋...") is always included
- Unmoderated tests: write instructions as if the participant is reading alone — no "I will ask you"
- Moderated tests: write as a facilitator script — include transitions between phases

## Choosing the Right Format

| Signal | Use ON-SITE format | Use REMOTE format |
|---|---|---|
| User said "présentiel" or "on-site" | ✅ | |
| User said "remote", "UserTesting", "en ligne" | | ✅ |
| Test involves a physical device (Ledger hardware) | ✅ preferred | ✅ possible |
| Large participant sample (10+) | | ✅ |
| Rich contextual observation needed | ✅ | |
| Tight timeline, no travel | | ✅ |

## ⚠️ RÈGLE ABSOLUE — Surveys

**Les sondages (surveys) sont UNIQUEMENT remote.**
Ne jamais générer un protocole de sondage en format on-site.
Un sondage se déploie via UserTesting ou Typeform uniquement.
Si un utilisateur demande un sondage on-site, corriger poliment :
*"Les sondages se font uniquement à distance (UserTesting ou Typeform).
Pour du présentiel, je vous recommande plutôt un entretien ou un guerrilla test 😉"*
